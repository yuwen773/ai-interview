# 小程序排版问题分析报告

## 1. 问题现象
当前小程序中的所有页面均出现了排版异常，主要表现为：
- 字体大小过大（如普通正文甚至标题看起来非常庞大）。
- 盒子（Box）的宽度、卡片（Card）的大小远超预期，显得非常拥挤甚至超出屏幕。
- 间距（Margin/Padding）过大，整体 UI 比例失调。

## 2. 核心原因分析
经过对项目代码的排查，根本原因在于 **Taro 的设计稿宽度配置（`designWidth`）与实际编写 CSS 时的尺寸标准不匹配**。

### 2.1 CSS 尺寸使用了 750px 视觉稿标准
项目中所有 `.scss` 样式文件中的像素（`px`）值，均是按照 **750px** 的设计稿标准来编写的。
例如：
- 在 `src/styles/theme.scss` 中，正文文本大小 `$font-size-body` 被定义为 `28px`，标题大小 `$font-size-title` 被定义为 `34px`。
- 在 `src/components/JobRoleCard/index.scss` 中，卡片内边距为 `28px`，标题字体大小为 `34px`，圆角为 `18px`。
- 此外还有如 `164px` 的卡片最小高度等设定。

在 750px 的设计稿中，`28px` 实际上对应移动端标准逻辑像素（Logical Pixels）的 `14px`（即正常正文字号），`34px` 对应 `17px`（正常标题字号）。

### 2.2 Taro 配置错误 (`designWidth: 375`)
虽然 CSS 中使用了 750px 的尺寸，但在 Taro 的全局编译配置文件 `config/index.js` 中，却将设计稿宽度配置为了 **375**：

```javascript
// 错误配置：config/index.js
const config = {
  // ...
  designWidth: 375,
  deviceRatio: {
    375: 2 / 1,
    // ...
  }
}
```

**Taro 的 px 转换机制：**
Taro 会在编译时自动将我们写的 `px` 转换为小程序的 `rpx` 单位。转换公式为：
`rpx = px * (750 / designWidth)`

由于 `designWidth` 被错误地设置为 `375`，Taro 在编译时会将所有的尺寸**乘以 2**：
- 代码中的 `28px`（本意是设计稿的 28 像素，屏幕上的 14逻辑像素）会被 Taro 转换为 `56rpx`。
- `56rpx` 在小程序中渲染出来的实际物理尺寸是 `28个逻辑像素`！
- 这直接导致了所有的字体、宽高、内外边距都比预期的**放大了一倍**，从而导致了页面排版完全崩溃、不适合小程序的情况。

## 3. 修复方案

只需要将 Taro 的编译配置修正为 750px 的设计稿标准即可解决此问题。

修改 `config/index.js` 中的 `designWidth` 和 `deviceRatio` 字段：

```javascript
// 修改后的 config/index.js
const config = {
  // ...
  designWidth: 750, // 修正为 750
  deviceRatio: {
    640: 2.34 / 2,
    750: 1,         // 750 对应 1:1 的 rpx 转换
    828: 1.81 / 2,
    375: 2 / 1
  },
  // ...
}
```

修改后，Taro 将采用 `1px = 1rpx` 的比例进行编译：
代码中的 `28px` 会被转换为 `28rpx`，在小程序中完美还原为移动端合适的 `14逻辑像素` 尺寸，所有的卡片、字体大小和盒模型排版将恢复正常的视觉比例。
