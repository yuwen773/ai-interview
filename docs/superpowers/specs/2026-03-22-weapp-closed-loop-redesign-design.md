# 微信小程序核心闭环重构设计

## 1. 背景与目标

当前 `uniapp-interview` 小程序存在三个核心问题：

1. 功能闭环不完整，无法充分承接 Web 端已有的求职训练能力。
2. 页面信息架构偏弱，首页、历史、详情之间缺乏清晰分工。
3. 视觉语言较弱，页面更像管理工具壳层，而不是面向校招用户的训练产品。

本次设计目标是将小程序第一阶段重构为一个面向校招用户的、可独立完成训练闭环的移动端产品，并严格以现有后端接口为边界设计；仅在聚合摘要场景中补充少量通用聚合接口。

## 2. 用户与范围

### 2.1 目标用户

- 主人群：校招和应届生
- 核心诉求：快速上传简历，进入模拟面试，获得清晰反馈，并看到练习进展

### 2.2 第一阶段范围

第一阶段聚焦以下完整闭环：

`上传简历 -> 查看解析结果 -> 选择目标岗位 -> 开始面试 -> 多轮作答 -> 生成报告 -> 查看历史报告 -> 再次练习 -> 查看成长曲线`

### 2.3 第一阶段不纳入范围

以下能力暂不作为小程序第一阶段重点：

- 复杂语音面试链路
- 3D 虚拟面试官
- 知识库管理
- 知识库问答助手
- 重度桌面工作台型交互
- 暗色模式

## 3. 设计原则

### 3.1 产品原则

- 优先闭环：先保证训练流程可独立完成。
- 优先复用：优先复用现有后端接口与业务能力。
- 优先移动端可用性：减少层级跳转和冗余操作，控制页面信息密度。
- 优先状态引导：首页承担下一步动作分发，而不是纯展示。

### 3.2 技术原则

- 页面与接口一一映射，避免设计依赖不存在的后端能力。
- 需要摘要统计时，由后端提供通用聚合接口，避免前端多次请求导致 N+1。
- 第一阶段不引入新的复杂交互基础设施。

## 4. 信息架构

### 4.1 Tab 结构

小程序第一阶段采用 `4 Tab + 6 流程页` 结构。

#### Tab 页面

1. `首页`
2. `简历库`
3. `面试记录`
4. `我的`

#### 非 Tab 流程页

1. `上传简历页`
2. `简历详情页`
3. `岗位配置页`
4. `面试进行页`
5. `面试报告页`
6. `成长曲线页`

### 4.2 用户主路径

#### 新用户

`首页 -> 上传简历 -> 简历详情 -> 岗位配置 -> 面试进行 -> 面试报告 -> 成长曲线`

#### 老用户

`首页 -> 继续未完成面试`

或：

`首页 -> 最近报告 -> 再次练习`

#### 重度用户

`简历库 -> 简历详情 -> 再次面试`

`面试记录 -> 查看报告 / 导出 / 删除`

## 5. 核心模块设计

### 5.1 首页

首页采用“上任务、下状态”的混合结构。

#### 页面职责

- 承担下一步动作分发
- 展示当前用户最重要的状态摘要
- 作为新用户与老用户的统一入口

#### 页面结构

1. 欢迎区
2. 任务卡区
3. 最近状态区
4. 快捷入口区

#### 动态任务卡逻辑

- 无简历：显示 `上传简历`
- 有简历但分析未完成：显示 `查看分析进度`
- 有未完成面试：显示 `继续面试`
- 有已完成报告：显示 `查看最近报告`

#### 快捷入口

- `简历库`
- `面试记录`
- `成长曲线`

#### 数据来源

首页不采用前端自行聚合多接口的方案，新增通用聚合接口。

建议新增：

`GET /api/dashboard/summary`

返回内容建议包含：

- `latestResume`
- `latestInterview`
- `latestReportScore`
- `resumeCount`
- `totalInterviewCount`
- `unfinishedInterviewCount`

### 5.2 简历库

#### 页面职责

- 管理简历资产
- 提供进入简历详情与再次练习的入口

#### 列表项内容

- 文件名
- 上传时间
- 分析状态
- 最新得分
- 面试次数

#### 页面主操作

- 上传新简历
- 查看简历详情
- 删除简历

#### 数据来源

直接复用：

`GET /api/resumes`

`DELETE /api/resumes/{id}`

### 5.3 面试记录

#### 页面职责

- 管理练习资产
- 继续未完成面试
- 查看历史报告与导出

#### 列表项内容

- 关联简历名
- 岗位
- 题目数
- 状态
- 分数
- 时间

#### 页面主操作

- 继续未完成
- 查看报告
- 导出 PDF
- 删除记录

#### 数据来源

记录列表不走前端跨简历循环拼装，建议新增通用聚合接口。

建议新增：

`GET /api/interview/history/summary`

返回内容建议包含：

- `stats.totalCount`
- `stats.completedCount`
- `stats.averageScore`
- `items[]`

其中每条 `items` 应包含：

- `sessionId`
- `resumeId`
- `resumeFilename`
- `jobRole`
- `jobLabel`
- `totalQuestions`
- `status`
- `evaluateStatus`
- `overallScore`
- `createdAt`
- `completedAt`

保留以下已有接口用于详情与操作：

`GET /api/interview/sessions/{sessionId}/details`

`GET /api/interview/sessions/{sessionId}/export`

`DELETE /api/interview/sessions/{sessionId}`

## 6. 流程页设计

### 6.1 上传简历页

#### 页面职责

- 选择文件
- 上传并触发分析
- 成功后进入简历详情

#### 接口映射

`POST /api/resumes/upload`

### 6.2 简历详情页

#### 页面职责

- 展示简历分析结果
- 展示该简历关联面试历史
- 提供重新分析、开始面试、导出 PDF 入口

#### 页面结构

1. 分析概览
2. 维度得分
3. 分析摘要与建议
4. 关联面试记录
5. 操作区

#### 接口映射

`GET /api/resumes/{id}/detail`

`POST /api/resumes/{id}/reanalyze`

`GET /api/resumes/{id}/export`

### 6.3 岗位配置页

#### 页面职责

- 选择目标岗位
- 展示当前简历摘要
- 检测是否有未完成会话
- 创建或恢复面试

#### 页面结构

1. 当前简历摘要
2. 岗位列表
3. 未完成面试提示
4. 开始按钮

#### 接口映射

`GET /api/interview/job-roles`

`GET /api/interview/sessions/unfinished/{resumeId}`

`POST /api/interview/sessions`

### 6.4 面试进行页

#### 页面职责

- 承载核心答题流程
- 支持提交、暂存、提前交卷
- 支持恢复未完成状态

#### 页面结构

1. 顶部进度条
2. 当前题目卡
3. 历史问答流
4. 文本输入区
5. 底部操作条

#### 第一阶段交互约束

- 以文本作答为主
- 暂不将复杂语音链路视为主能力
- 保留后续扩展位，但不影响当前页面主路径

#### 接口映射

`GET /api/interview/sessions/{sessionId}`

`POST /api/interview/sessions/{sessionId}/answers`

`PUT /api/interview/sessions/{sessionId}/answers`

`POST /api/interview/sessions/{sessionId}/complete`

### 6.5 面试报告页

#### 页面职责

- 展示训练结果
- 呈现数据反馈
- 提供导出与再次练习入口

#### 页面结构

1. 总分 Hero
2. 分类得分条
3. 总体评价
4. 优点总结
5. 改进建议
6. 逐题反馈折叠列表
7. 参考答案 / 关键点
8. 导出 PDF

#### 接口映射

`GET /api/interview/sessions/{sessionId}/report`

`GET /api/interview/sessions/{sessionId}/export`

#### 报告数据粒度

后端已有 DTO 已能支撑以下展示：

- `overallScore`
- `categoryScores`
- `questionDetails`
- `overallFeedback`
- `strengths`
- `improvements`
- `referenceAnswers`

### 6.6 成长曲线页

#### 页面职责

- 展示训练后的进步趋势
- 提供继续练习的激励

#### 页面结构

1. 趋势折线图
2. 岗位切换器
3. 最近成绩摘要

#### 设计要求

- 小程序采用轻量版趋势图
- 不照搬 Web 宽屏多列布局
- 以“是否进步”作为核心认知点

#### 接口映射

`GET /api/interview/growth?resumeId=...`

## 7. 后端接口策略

### 7.1 直接复用的现有接口

#### 简历侧

- `POST /api/resumes/upload`
- `GET /api/resumes`
- `GET /api/resumes/{id}/detail`
- `POST /api/resumes/{id}/reanalyze`
- `GET /api/resumes/{id}/export`
- `DELETE /api/resumes/{id}`

#### 面试侧

- `GET /api/interview/job-roles`
- `POST /api/interview/sessions`
- `GET /api/interview/sessions/{sessionId}`
- `GET /api/interview/sessions/unfinished/{resumeId}`
- `POST /api/interview/sessions/{sessionId}/answers`
- `PUT /api/interview/sessions/{sessionId}/answers`
- `POST /api/interview/sessions/{sessionId}/complete`
- `GET /api/interview/sessions/{sessionId}/report`
- `GET /api/interview/sessions/{sessionId}/details`
- `GET /api/interview/sessions/{sessionId}/export`
- `DELETE /api/interview/sessions/{sessionId}`
- `GET /api/interview/growth`

### 7.2 新增的通用聚合接口

仅在摘要统计场景增加通用聚合接口，以减少小程序和 Web 的多次请求与多次数据库 IO。

#### 接口一：首页摘要

`GET /api/dashboard/summary`

适用端：

- Web
- 微信小程序

#### 接口二：面试记录摘要

`GET /api/interview/history/summary`

适用端：

- Web
- 微信小程序

### 7.3 不建议的方案

不建议采用以下方式：

- 前端调用 `/resumes` 后再逐个调用 `/resumes/{id}/detail`
- 前端循环拼接面试记录和统计
- Web 和 weapp 各自维护不同统计口径

原因：

- 请求次数多
- 数据口径不一致
- 小程序首屏性能更差
- 后端更适合承担聚合视图职责

## 8. 视觉系统

### 8.1 总体方向

视觉方向定义为：

`清爽但有张力的学习训练风格`

目标不是做成后台管理界面，也不是做成过度炫技的概念风，而是一个面向校招用户的训练产品。

### 8.2 色彩策略

#### 主色

`深海军蓝`

用于：

- 顶部核心区域
- 主标题
- 关键结构层

#### 强调色

`电光青蓝`

用于：

- 选中态
- 进度
- 局部数据强调

约束：

- 只做强调色，不做大面积背景

#### 激励色

`暖橙`

用于：

- CTA
- 成绩高亮
- 成就反馈

约束：

- 不做基础背景
- 不与大面积冷色直接冲突

### 8.3 排版规则

- 主标题、分数数字、关键 CTA 强化字重与层级
- 正文说明克制，减少“所有信息一个层级”的问题
- 统一数字展示语言

建议视觉规格：

- 分数数字：`48px` 加粗，单位小字
- 统计数字：`24px - 32px` 中粗
- 标签文字：`12px`
- 状态标签：胶囊样式

### 8.4 卡片体系

#### 任务卡

特点：

- 强背景对比
- 更明显的图标区
- 用于驱动下一步动作

建议圆角：`16px`

#### 信息卡

特点：

- 白底或浅灰底
- 克制边框
- 用于承载最近状态与基础信息

建议圆角：`14px`

#### 结果卡

特点：

- 强调数字与进度条
- 用于报告、分数、成长曲线

建议圆角：`12px`

### 8.5 动效策略

第一阶段仅使用三类动效：

1. 页面进入轻微上浮
2. 卡片点击压缩反馈
3. 分数与进度条加载动画

建议参数：

- 页面进入：`translateY(8px) -> 0`，`300ms ease-out`
- 点击压缩：`scale(0.97)`
- 进度条动画：`0 -> target`，`800ms ease-out`

### 8.6 各页面气质定义

- 首页：训练面板，强调“下一步做什么”
- 简历详情：分析档案，强调“当前状态 + 行动”
- 面试进行页：专注答题，视觉收敛
- 报告页：成果展示，视觉最强
- 成长曲线页：进步反馈，激励继续练习

### 8.7 暗色模式策略

第一阶段不做暗色模式。

原因：

- 优先把浅色体系做扎实
- 减少设计与开发变量
- 避免强调色在暗色下失控

## 9. 错误处理与状态设计

### 9.1 必需状态

所有关键页面需统一考虑以下状态：

- Loading
- Empty
- Error
- Processing
- Completed

### 9.2 关键异常场景

- 上传失败
- 简历分析处理中
- 简历分析失败
- 创建面试失败
- 提交答案失败
- 报告生成中
- 报告生成失败
- 成长曲线为空

### 9.3 处理原则

- 错误提示使用可执行文案
- 尽量提供重试动作
- 不因单一异步失败阻塞整个主流程

## 10. 测试与验证要求

### 10.1 手动验证范围

第一阶段至少验证以下主流程：

1. 新用户上传简历并完成一次面试
2. 老用户恢复未完成面试
3. 查看报告并导出 PDF
4. 查看简历详情并重新分析
5. 查看成长曲线
6. 查看面试记录并删除记录

### 10.2 兼容性要求

- 微信小程序主流机型
- 页面首屏与滚动流畅度
- 长文本报告的可读性
- 异步处理中断后的恢复能力

## 11. 风险与约束

### 11.1 已识别风险

- 当前小程序存在中文编码异常，影响真实体验与文案校验
- 现有页面样式已被改动，后续实施需谨慎与用户现有改动并行
- 若不新增聚合接口，首页与记录页会出现多请求性能问题

### 11.2 实施约束

- 不回退用户已有本地改动
- 不把知识库与桌面工作台型功能强行塞进小程序第一阶段
- 所有新页面与数据视图应优先复用现有接口与 DTO

## 12. 结论

本设计选择 `闭环重构型` 路线：

- 以校招用户为主
- 以核心训练闭环为先
- 以现有后端接口为边界
- 仅补充少量通用聚合接口
- 以清爽但有张力的训练产品视觉作为统一语言

该方案既能补齐当前小程序严重不足的能力缺口，也能为第二阶段品牌体验强化保留充足空间。
