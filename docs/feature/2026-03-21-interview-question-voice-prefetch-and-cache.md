# 模拟面试题目语音预取与本地缓存设计

## 1. 文档目的

本文档用于定义模拟面试题目语音播报的下一阶段优化方案，目标是在不改变当前面试主流程的前提下，提升题目语音播报的响应速度、重复播放体验和整体稳定性。

本文档只覆盖 Web 端 `frontend/` 与现有后端 TTS 接口协作，不引入服务端音频持久化。

## 2. 当前现状

当前题目播报链路如下：

1. 面试会话创建时，一次性生成完整题单
2. 用户提交当前题答案后，前端拿到下一题文本
3. 若开启 `textVoice`，前端调用 `POST /api/interview/tts/stream`
4. 后端即时合成音频并通过 SSE 返回
5. 前端将音频块拼成 `Blob`，使用 `URL.createObjectURL()` 播放

当前实现特点：

1. 题目是预生成题单，适合做预取
2. TTS 是按需即时生成，不做服务端缓存
3. 前端已有页面内临时缓存，但仅在当前页面生命周期内有效
4. 页面刷新、重新进入会话后，重复播放同一题仍会重新请求 TTS

## 3. 问题定义

当前方案存在以下体验问题：

1. 用户进入新题后，首次语音播报存在等待时间
2. 重复播放已听过的题目时，缓存命中范围过小
3. 语音播报链路完全依赖“播放时即时拉取”，抗抖动能力一般

同时，当前项目暂不适合直接做服务端持久化音频：

1. 题目内容高度个性化，跨用户复用率有限
2. 引入 MinIO + MySQL 音频元数据会增加复杂度
3. 一期主目标仍然是提升 Web 端播报体验，而不是建设通用音频资产中心

## 4. 设计目标

本次优化目标：

1. 在开启语音播报模式时，提前为后续题目做 TTS 预取
2. 将预取结果存入浏览器本地持久缓存，提高重复播放命中率
3. 不改变现有 `submitAnswer()` 主业务链路
4. 不阻断文字面试主流程
5. 以较低实现复杂度获得明显体验收益

## 5. 非目标

本次不做：

1. MinIO 持久化题目音频
2. MySQL 存储 TTS 音频元数据
3. 全量题目一次性预生成音频
4. 语音参数配置中心
5. 多端共享音频缓存

## 6. 核心方案

### 6.1 总体思路

采用“两级缓存 + 滑动窗口预取”：

1. 一级缓存：当前页面内存缓存
2. 二级缓存：IndexedDB 持久缓存
3. 预取策略：仅在 `textVoice` 模式下，对当前题之后的若干题进行后台预取

### 6.2 为什么不建议一次性预取全部题目

虽然题单是预生成的，但一次性预取全部题目并不是最优解，原因如下：

1. 用户可能中途退出，产生无效 TTS 请求
2. 首次进入会话的后台开销变大
3. 浏览器本地缓存膨胀更快
4. 后续题目不一定都会被真正播放

因此更合理的方式是滑动窗口预取，而不是全量预取。

## 7. 预取策略设计

### 7.1 推荐策略

推荐采用窗口大小为 `2` 或 `3` 的滑动预取。

默认建议：

1. 当前题进入可播报状态时，优先确保当前题音频已可播放
2. 后台预取后续 `2` 题
3. 当用户推进到第 `i` 题时，后台补齐到第 `i + 2` 题

示例：

1. 当前位于第 1 题时，缓存目标为第 1、2、3 题
2. 当前推进到第 2 题时，检查并补齐第 4 题
3. 当前推进到第 3 题时，检查并补齐第 5 题

### 7.2 触发条件

仅在满足以下条件时启动预取：

1. 面试官输出模式为 `textVoice`
2. 当前会话存在有效题单
3. 浏览器支持 IndexedDB
4. 当前没有被显式停止预取

### 7.3 停止条件

出现以下情况时停止预取：

1. 用户切换到 `text` 模式
2. 会话结束
3. 页面卸载
4. 当前预取任务被新的预取任务覆盖

### 7.4 并发控制

建议：

1. 预取请求串行执行，或最大并发数限制为 `2`
2. 当前题播放请求优先级高于后台预取
3. 同一题只允许一个在途 TTS 请求

## 8. 缓存设计

### 8.1 缓存分层

#### 一级缓存：内存缓存

继续保留当前 `useQuestionVoicePlayer` 中的页面级缓存，优点是读取快、实现简单，适合：

1. 当前题重复播放
2. 同一页面会话内快速重播

#### 二级缓存：IndexedDB

新增 IndexedDB 持久缓存，优点是：

1. 页面刷新后仍可命中
2. 同一用户在当前浏览器中重复进入会话时仍可复用
3. 不需要后端持久化改造

### 8.2 缓存 Key 设计

建议缓存键包含以下维度：

1. `sessionId`
2. `questionIndex`
3. `textHash`
4. `voiceParamsHash`

原因：

1. `sessionId + questionIndex` 便于按会话清理
2. `textHash` 避免同索引内容变化时误命中
3. `voiceParamsHash` 避免音色、采样率、语速调整后误复用旧音频

推荐键格式：

```text
interview-question-voice:{sessionId}:{questionIndex}:{textHash}:{voiceParamsHash}
```

### 8.3 缓存 Value 设计

建议存储：

1. `audioBlob` 或 `ArrayBuffer`
2. `mimeType`
3. `text`
4. `createdAt`
5. `lastAccessedAt`
6. `size`

### 8.4 容量控制

IndexedDB 不能无限增长，建议增加轻量治理策略：

1. 仅缓存最近若干个活跃 session
2. 设置总容量上限，例如 `30MB`
3. 超出上限时按 `lastAccessedAt` 做 LRU 淘汰

## 9. 播放与预取流程

### 9.1 播放流程

当页面需要播放当前题目时：

1. 先查内存缓存
2. 未命中则查 IndexedDB
3. 仍未命中则请求后端 TTS
4. 请求成功后写入 IndexedDB
5. 同时写入内存缓存并播放

### 9.2 预取流程

当用户开启 `textVoice` 且题目变更时：

1. 计算滑动窗口内需要存在缓存的题目集合
2. 对每道题按顺序检查内存缓存和 IndexedDB
3. 只对未命中的题目发起 TTS 请求
4. 请求成功后写入 IndexedDB
5. 不自动播放预取结果

### 9.3 异常兜底

若预取失败：

1. 不提示阻断性错误
2. 仅记录日志或埋点
3. 真正播放时再走正常播放链路重试

若当前题实时播放失败：

1. 提示“题目语音播放失败”
2. 不影响文字题目展示和答题推进

## 10. 前端实现建议

### 10.1 建议新增模块

建议新增以下文件：

1. `frontend/src/utils/interviewVoiceCache.ts`
2. `frontend/src/hooks/useQuestionVoicePrefetch.ts`

必要时可新增类型文件：

1. `frontend/src/types/interviewVoice.ts`

### 10.2 模块职责

#### `interviewVoiceCache.ts`

职责：

1. IndexedDB 初始化
2. 音频读写
3. 缓存键生成
4. 容量清理
5. session 级清理

建议接口：

```ts
getQuestionAudio(params): Promise<Blob | null>
setQuestionAudio(params, blob): Promise<void>
deleteSessionAudio(sessionId): Promise<void>
trimCacheIfNeeded(): Promise<void>
```

#### `useQuestionVoicePrefetch.ts`

职责：

1. 监听题目索引和输出模式变化
2. 根据滑动窗口触发后台预取
3. 控制取消、去重与并发

### 10.3 对现有 `useQuestionVoicePlayer` 的改造

建议改造方向：

1. 播放前先查 IndexedDB
2. TTS 请求成功后写入 IndexedDB
3. 当前 `objectUrlRef` 继续作为一级缓存
4. 保持现有 `stopPlayback()` 语义不变

## 11. 后端协作要求

本方案尽量不改后端接口，仅复用现有：

1. `POST /api/interview/tts/stream`

后端当前按文本即时合成并返回音频，已满足本次方案要求。

后端可选优化：

1. 补充 TTS 请求耗时、成功率埋点
2. 增加请求文本长度与响应大小日志
3. 如后续需要，再引入服务端短期缓存

## 12. 数据一致性与风险

### 12.1 一致性

由于题目列表在会话创建时已经固定，当前方案具备较好的一致性基础：

1. 题目文本通常不会在会话中途变化
2. 同一题的语音结果可安全复用

### 12.2 风险点

主要风险如下：

1. 预取请求过多，影响当前播放体验
2. IndexedDB 容量膨胀
3. 同题重复请求导致资源浪费
4. 浏览器隐私模式或受限环境下 IndexedDB 不稳定

对应控制措施：

1. 限制预取窗口和并发数
2. 增加 LRU 清理
3. 为 in-flight 请求做去重
4. IndexedDB 不可用时自动降级为内存缓存

## 13. 验收标准

满足以下条件可视为方案落地成功：

1. 开启 `textVoice` 后，当前题播报平均等待时间明显下降
2. 已播放题目刷新页面后仍可重复播放
3. 关闭 `textVoice` 后不再继续后台预取
4. 预取失败不影响文字面试主流程
5. 本地缓存不会无限增长

## 14. 实施顺序建议

建议按以下顺序实施：

### 阶段 1：IndexedDB 能力接入

1. 新增本地缓存读写工具
2. `useQuestionVoicePlayer` 接入读取与写入
3. 验证重复播放和刷新后复播

### 阶段 2：滑动窗口预取

1. 新增 `useQuestionVoicePrefetch`
2. 接入面试页面状态变化
3. 增加去重、取消和并发限制

### 阶段 3：治理与埋点

1. 增加缓存容量控制
2. 增加预取命中率与失败率埋点
3. 补充异常场景验证

## 15. 最终结论

对于当前项目阶段，最优解不是直接做服务端持久化音频，而是：

1. 基于预生成题单做滑动窗口 TTS 预取
2. 使用 IndexedDB 作为浏览器本地持久缓存
3. 保留当前页面内存缓存作为一级缓存

这套方案的特点是：

1. 收益明确
2. 对现有业务链路侵入小
3. 技术复杂度可控
4. 便于后续平滑升级到更强的缓存体系
